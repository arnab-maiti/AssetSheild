# 🚀 AssetShield

Dynamic NFT-based Real World Asset Verification System

---

## 📌 Overview

AssetShield is a blockchain-integrated system designed to verify, track, and manage real-world assets (documents, certificates, licenses) using NFTs as a proof layer.

The goal is to prevent fraud, ensure authenticity, and provide a transparent lifecycle of assets.

---

## 🧠 System Architecture

Frontend → Backend → Database → IPFS → Blockchain → ML Service

### Key Roles:
- **Backend** → Core logic, validation, orchestration
- **Database** → Source of truth
- **Blockchain** → Immutable proof layer (NFT)
- **IPFS** → Decentralized metadata storage
- **ML Service** → Risk analysis & fraud detection (async)

---

## 🧩 Core Design Decisions

### 1. NFT ≠ Ownership
NFT represents proof of issuance, NOT legal ownership.

### 2. Dual Verification System
Verification is done using:
- Backend validation (DB + logic)
- Blockchain proof (NFT existence)

### 3. Secure QR Verification
- Generated by backend
- Linked to verification API
- Prevents tampering

### 4. Async ML Integration
- ML runs independently
- System does NOT depend on ML to function
- Used for risk scoring and anomaly detection

### 5. Authority Control
- Wallet-based identity
- Controlled via backend + smart contract

## 🗄️ Database Design

### Core Tables

#### authorities
Stores authorized issuers

#### assets
Stores asset metadata
- document_hash
- status (active / revoked / expired)

#### nft_tokens
Maps assets to blockchain NFTs
- token_id
- contract_address

#### asset_events ⭐
Tracks every action in asset lifecycle
- mint
- verify
- revoke
- update

👉 **Backbone of the system**

#### risk_scores
Stores ML-based risk analysis
- similarity_score
- is_duplicate
- is_suspicious

#### verification_logs
Logs every verification request
- verification_url
- timestamp

---

## 🔁 System Flows

### 1. Mint Flow
Upload → Hash → Store (IPFS) → Save in DB → Mint NFT

### 2. Verify Flow
API Request → DB Check → Blockchain Verification → Response

### 3. Revoke Flow
Update Status → Log Event → Reflect in Verification

### 4. ML Flow (Async)
Event Logs → ML Service → Update Risk Scores

---

## 🔐 Security Design

- SHA-256 hashing for tamper detection
- No reliance on OCR for authenticity
- Wallet-based authority verification
- Immutable event logging
